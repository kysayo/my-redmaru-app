# 引き継ぎ: AI回答の自動送信・待機・書き戻し機能

別セッション（view-customizeプロジェクト）での検討内容の引き継ぎ。ここに書かれている前提を踏まえてmy-redmaru-app側の実装を進めてほしい。

## 背景・ゴール

現状の「to marucha」機能（`redmine.content.ts` → `background.ts` → `aichat.content.ts`）は、チケット情報をAIチャットに貼り付けるところまでで、送信・回答待ち・Redmineへの書き戻しは人間がやっている。これを自動化したい。

作りたい機能は2種類。実装するのは主に(1)で、(2)は将来的な別アプリ（このリポジトリの外）。

1. **単発ボタン（拡張機能本体）**: チケット詳細ページのボタンを押すと、AIチャットへの送信・回答待ち・Redmineへの書き戻しまで自動で完結する。1件だけ今すぐ結果が欲しい時用。
2. **Playwrightバッチ（将来・別アプリ）**: Node実行環境からPlaywrightでこの拡張機能入りのChromeを操作し、鮮度が古いチケットを一括で処理する。(1)のボタンをプログラムから機械的にクリックするだけなので、自動化ロジック自体は(1)を再利用する想定。

## (1) 単発ボタンの自動化フロー

既存の流れ（`redmine.content.ts`のボタン→`background.ts`→`aichat.content.ts`でテキスト挿入）に以下を追加する。

1. **自動送信**: テキスト挿入後、送信ボタンをクリック（またはEnter dispatch）。セレクタは実サイト（`https://www.marubeni-chatbot.com`）をdevtoolsで確認して決める。
2. **回答完了の検知**: `MutationObserver`で回答エリアの変化を監視し、一定時間（1.5〜2秒程度）DOM変化が止まったら完了とみなすデバウンス方式。理想は「生成中」を示すUI要素（送信ボタンのdisabled/enabled切り替え、停止ボタンの出現・消失など）と組み合わせる。**必ずタイムアウト上限（例: 90秒）を設け、検知失敗時は書き込みをせず諦める**（誤って古い/中途半端な回答を書き込む方が待たされるより害が大きいため）。
   - 完了検知のセレクタは実サイトを見ないと分からないので、devtoolsでの確認・試行錯誤が前提。`maruchat-focus.content.ts`の`.segment-based-content`が回答テキストのクラスとして既に使われている（`openIsouForm`参照）ので、ここが手がかりになる。
3. **Redmineへの書き戻し**: ISOU申請のように別タブを開き直してDOM操作する必要はない。Redmineは既にAPIキーを持っている（`redmine-bridge.content.ts`経由で`ViewCustomize.context.user.apiKey`を取得済み）ので、`fetch`で直接`PUT /issues/{id}.json`すればよい。元のRedmineタブが閉じていても無関係に動く。

### 書き戻しフィールドの契約（view-customize側と合わせ済み）

- `cf_4588`（AI更新日時、テキスト型）: フォーマットは **`YYYY-MM-DD HH:mm:ss`**（例: `2026-07-01 13:45:00`）。Redmine側カスタムフィールドの正規表現バリデーションに以下を設定済み/設定予定：
  `^\d{4}-(0[1-9]|1[0-2])-(0[1-9]|[12]\d|3[01]) ([01]\d|2[0-3]):[0-5]\d:[0-5]\d$`
- `cf_4589`（AI回答、長いテキスト型）: AIの回答テキストそのもの。
- **重要**: `cf_4588`と`cf_4589`は**同一のPUTリクエストで同時に書き込むこと**。理由: view-customize側の鮮度判定（後述）は「チケットの`updated_on`」と「`cf_4588`の値」を比較するため、別々のタイミングで書くと書き戻し直後から誤って「古い」と判定されてしまう。同時に書けば`updated_on == cf_4588`（ほぼ一致）になり、以降に人間が更新した場合だけ正しく「古い可能性あり」の警告が出る。

### 品質担保についての合意事項

数が多いため、送信前後の人間による確認ゲートは**設けない**方針で合意済み（ユーザーの明示的な判断）。その代わりview-customize側の表示（後述）で「AIの自動回答であり誤りを含みうる」旨と「回答後の更新有無」を常時可視化することで、読み手側が判断できるようにする設計。

## (2) Playwrightバッチ（将来・別アプリ）についてのメモ

- 拡張機能はChrome Web Store配布ではなく、**ローカルのunpackedビルド（WXTの出力フォルダ、例: `.output/chrome-mv3`）を直接指定する**方式を想定。Playwright側は`launchPersistentContext(userDataDir, { args: ['--disable-extensions-except=<path>', '--load-extension=<path>'] })`で読み込む。
- MV3拡張はheadlessモードで動作しないことがあるため、`headless: "new"`（対応バージョン要確認）かheadedで動かす前提で検討する。
- **専用のuserDataDirを使うこと**。ユーザーの普段使いのChromeプロファイルと共有すると、Chromeの排他ロックで競合する。専用プロファイルで一度だけ手動ログインしておけば、SAML系のセッションCookieが永続化されて以降のPlaywright起動でも再利用される（はず）。
- SSOの自動ログイン（ID/パスワード入力なしでボタン一つで通る仕組み）の正体が未確認。単なる永続Cookie・IdPセッションであれば無人実行でも問題なく通り続けるはずだが、Windows Hello/スマートカードのような人間の存在が前提の仕組みだった場合は無人実行が破綻する。**実際に専用プロファイルで手動ログイン→無人（スケジュール）実行、を試して検証する必要がある**。
- 「どのチケットが鮮度切れか」の抽出は、ブラウザを開かずともRedmine REST APIだけで判定可能（`cf_4589`が空でない、かつ`cf_4588`から計算した鮮度が閾値超え）。view-customize側の`src/script_05.txt`と同じロジックをNode側でAPI直叩きで再現すればよい。対象チケットのみブラウザで開いて拡張機能のボタンをクリックする、という流れにすれば無駄がない。

## 調査済み事項（AIチャット側のAPI直叩きは不可と判断）

View Customizeスクリプト単体（ブラウザ拡張なし）でRedmineページから直接AIチャットのAPIを叩けないか検討したが、devtoolsのNetworkタブで通常のチャット送信を確認したところ分かりやすいPOSTエンドポイントが見当たらず、認証トークンも長く、無許可でクロスオリジン接続を試すのは避けるべきと判断して断念した。よってブラウザ拡張のcontent script経由でのDOM自動操作が唯一の現実的な経路になる。

## 対応するview-customize側の実装（別リポジトリ、実装済み）

`E:\Projects\view-customize\src\script_05.txt`にて、チケット詳細ページで`cf_4589`に値があれば免責文言を表示し、`cf_4588`とチケットの正確な`updated_on`（DOM表示は分単位に丸められているため使わず、`GET /issues/{id}.json`のREST APIから取得）を60秒の許容誤差で比較して鮮度警告を表示する機能を実装済み。これは拡張機能の有無に関係なく動く独立機能（Redmineのチケットはチーム外ユーザーも閲覧するため）。

実際のRedmine（misol-dev.cloud.redmine.jp）のDOM構造も確認済み：
- カスタムフィールド: `<div class="string_cf cf_4588 attribute">` / `<div class="text_cf cf_4589 attribute">`、内部に`<div class="label">`と`<div class="value">`
- チケットヘッダーの更新日時表示: `<p class="author">`内の`<a title="YYYY/MM/DD HH:mm" data-absolute-date="...">`（分単位、秒がないためAPI併用が必須だった）

### 実機確認で当初の想定から変わった点

1. **送信方法**: 当初は「送信ボタンをクリック（またはEnter dispatch）」を想定していたが、実機では**送信ボタンの`click()`だけでは送信されないケースがあった**（フレームワーク側の内部状態更新が、DOM上の値変更・disabled解除に追いついていない模様）。実サイトの入力欄プレースホルダーが「Ctrl + Enterキーを押して送信」と明記していたため、Ctrl+Enterのキーイベントをシミュレートする方式に変更し、これで送信できることを確認した。ボタンクリックは併用フォールバックとして残している。
2. **完了検知**: 想定していた「回答エリアのDOM変化debounce」に加えて、実機で「この回答は役に立ちましたか？」というフィードバックアンケートの出現を主判定として採用した。ただしこのアンケート要素は`class="hidden ..."`を持ち、**生成完了前からCSSで非表示のままDOMに存在している場合がある**ことが実機テストで判明したため、「存在有無」ではなく「監視開始時点をベースラインとした個数の増加」で判定する方式に変更した（そうしないと送信直後に誤って完了判定してしまい、生成途中の内容がRedmineに書き込まれる事故が実際に1度発生した）。
3. **MV3 Service Workerのメッセージチャンネル**: 回答完了待ち（最大90秒）をcontent script側の非同期関数内で行い、その関数を`onMessage`リスナーが`await`する実装にしていたところ、待機中にService Workerが休止・再起動すると「メッセージチャンネルが応答前に閉じられた」というエラーが発生することが分かった（実害はないがエラーログが不要に出る）。応答を必要としないメッセージなのでリスナーを同期関数にしてfire-and-forgetにすることで解消した。
4. **書き戻し後のページ反映**: 当初特に取り決めていなかったが、Redmineのカスタムフィールド表示はサーバーレンダリングのため自動では更新されない。90秒の待機中にユーザーが同じタブで別作業（コメント入力等）をしている可能性を考慮し、自動リロードはせず、完了後にボタンラベルを「更新完了（クリックで再読込）」に変えてユーザーが任意のタイミングでリロードできるようにした。

### 未解決・調整中の課題

- **稀に「メッセージの送信に失敗しました。時間をおいて再度試してください」が発生する**: テキスト挿入直後にCtrl+Enterを発火するタイミングが早すぎることが原因と見られる。対策として、送信ボタンのdisabled解除をポーリングで待った上で、さらに300ms・800msの追加猶予を挟む変更を加えたが、これで完全に解消したかは2026-07-08時点で継続検証中。再発する場合は猶予時間のさらなる調整が必要。
- 生成中インジケーター（停止ボタン等）の実セレクタは依然未確認（`entrypoints/shared/aichatDom.ts`の`SELECTORS.stopButton`はプレースホルダのまま）。完了検知は上記のアンケート差分方式で代替できているため必須ではないが、精度を上げる余地として残っている。
